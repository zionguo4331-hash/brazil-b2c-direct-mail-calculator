# 诚达运通 PRC 报价提炼准确性验证样例

## 1. 这份文档是干什么的

这份文档用于验证：

1. 系统是否正确识别了产品代码
2. 系统是否正确识别了主运费
3. 系统是否正确识别了处理费重量段
4. 系统是否正确识别了尾程区域和邮编匹配
5. 系统是否正确把主运费 + 处理费 + 尾程费组合成总物流成本

这不是让你检查每个字段，而是用几组真实样例做“回算验证”。

## 2. 本次验证使用的原始报价文件

文件名：

`诚达运通巴西小包PRC专线报价4.24客户版(1).xlsx`

主要引用的 3 个 Sheet：

1. `PRC专线报价`
2. `尾程派送报价`
3. `商业清关区域划分`

## 3. 本次验证默认前提

除非某个样例另有说明，否则统一使用下面这些设置：

1. 选择同一张已提炼出的诚达运通 PRC 报价卡
2. `CNY → USD = 0.1468`
3. 尺寸：
   - 长 `20`
   - 宽 `15`
   - 高 `10`
4. 体积重除数：`6000`

由此可得：

- 体积重 = `20 × 15 × 10 / 6000 = 0.50 kg`

所以：

- 如果实重小于 `0.50kg`，计费重应取 `0.50kg`
- 如果实重大于 `0.50kg`，计费重应取实重

## 4. 你应该重点验证哪些结果

在系统里，优先核对下面这些内容：

1. 识别出的产品代码是否正确
2. CEP 是否匹配到正确区域
3. 计费重是否正确
4. 主运费是否正确
5. 处理费是否命中正确重量段
6. 尾程费是否命中正确区域和重量段
7. 总物流成本是否正确

如果这 7 项都对上，基本可以判断“提炼结果是可信的”。

## 5. 验证样例 1：体积重大于实重

### 你在页面里填写

1. 产品代码：`BR1001`
2. 实重：`0.40`
3. 尺寸：`20 × 15 × 10`
4. CEP：`01254-999`

### 系统应该识别到

1. 区域：`SP-INT2`
2. 计费重：`0.50kg`

原因：

- 实重 `0.40kg`
- 体积重 `0.50kg`
- 计费重取较大值，所以应为 `0.50kg`

### 你应该得到的答案

1. 主运费：
   - `80 × 0.50 = 40.00 CNY`
2. 处理费：
   - 重量段 `0.050-0.50`
   - `28.82 CNY`
3. 尾程费：
   - 区域 `SP-INT2`
   - 重量段 `0.251-0.50`
   - `32.81 CNY`
4. 总物流成本：
   - `40.00 + 28.82 + 32.81 = 101.63 CNY`
5. 按 `0.1468` 换算后：
   - `101.63 × 0.1468 = 14.9193 USD`

### 这一组主要验证什么

1. 体积重逻辑
2. CEP 匹配逻辑
3. 尾程矩阵 0.251-0.50 这一档是否取对

## 10. 如何判断“提炼基本准确”

如果下面这几项都对得上，说明提炼结果基本可信：

1. 5 组样例里，区域匹配都正确
2. 产品代码 `BR1001 / BR1002` 主报价都正确
3. 处理费重量段都切对
4. 尾程矩阵重量段都切对
5. 总物流成本只存在极小的四舍五入差异

## 11. 如果对不上，优先怀疑哪里

如果结果不对，优先检查：

1. 产品代码是否选错
2. CEP 是否输错
3. 是否命中了错误的重量段
4. 是否把实重和体积重搞反了
5. 是否尾程费没有正确取到对应区域
6. 汇率是不是不是 `0.1468`

## 12. 最后建议

建议你至少先跑这 5 组。

如果其中 4 组以上都能严格对上，基本说明：

1. 报价提炼是靠谱的
2. CEP 匹配是靠谱的
3. 主报价 + 处理费 + 尾程费的组合逻辑是靠谱的
